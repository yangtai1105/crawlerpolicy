# Plan 2: Static Site ‚Äî Implementation Outline

**Goal:** An Astro static site that reads Plan 1's output (`content/events/*.md`, `content/snapshots/*/*.html`, `data/*.json`, `sources.yaml`) and renders the A+C hybrid homepage plus per-event, per-source, about, subscribe, and feed.xml routes. Served locally via `astro dev` for immediate preview; ready to deploy to Vercel.

**Tech stack:** Astro 4, TypeScript, plain CSS with custom properties (no framework), self-hosted display font via `@fontsource-variable`, `js-yaml` for sources.yaml parsing, `reading-time` optional. No React / Vue.

**Architecture:**
- `site/` lives at repo root; `astro dev` runs from there.
- Content collections point at `../content/events/` and `../data/*.json`.
- Fully SSG ‚Äî `astro build` produces a static `site/dist/`.
- Aesthetic: **editorial, dark default** (The Information / Platformer vibe). Display serif for titles, variable sans for body, monospace for diffs. Information-dense.

---

## Phase 0 ‚Äî Scaffold `site/`

- **0.1** Run `npm create astro@latest site -- --template minimal --typescript strict --no-install --no-git` from repo root, install deps with `npm install`.
- **0.2** `astro.config.mjs`: `output: "static"`, `outDir: "./dist"`. Add `content/` collections.
- **0.3** Add deps: `js-yaml`, `@types/js-yaml`, `@fontsource-variable/inter`, `@fontsource/fraunces`.

## Phase 1 ‚Äî Content loaders

- **1.1** `src/content/config.ts` ‚Äî define `events` collection reading `../content/events/*.md` with Zod schema matching frontmatter (slug, title, source, pillar, detected_at, source_url, change_kind, importance).
- **1.2** `src/lib/sources.ts` ‚Äî load + parse `../sources.yaml`, expose typed `Source[]` + helpers (`findSource(slug)`, `crawlerSources()`).
- **1.3** `src/lib/sop.ts` ‚Äî load `../data/opt-out-matrix.json` with TS type.
- **1.4** `src/lib/snapshots.ts` ‚Äî helper to list snapshot files per source for `/sources/[slug]`.

## Phase 2 ‚Äî Base layout + styles

- **2.1** `src/layouts/Base.astro` ‚Äî `<html>` root, meta, font preload, nav (site title + links), footer (`data/health.json` last-run timestamp + LLM-disclosure fine-print).
- **2.2** `src/styles/global.css` ‚Äî CSS custom properties (color tokens, font stack, spacing scale), base element styles, editorial density defaults. Dark by default; light theme via `[data-theme="light"]`.
- **2.3** Components in `src/components/`:
  - `PillarTag.astro` ‚Äî color-coded chip (crawler=amber, ecosystem=cyan, agent=green).
  - `SourceBadge.astro` ‚Äî small "source ¬∑ time ago" line.
  - `TimeAgo.astro` ‚Äî server-rendered relative time.
  - `EventCard.astro` ‚Äî feed card (used on homepage + /sources).
  - `HeroBlock.astro` ‚Äî dark hero (top-importance event).
  - `StateOfPlay.astro` ‚Äî right-sidebar widget.
  - `ThemeToggle.astro` ‚Äî client-side dark/light switch.

## Phase 3 ‚Äî Homepage (`/`)

- **3.1** Fetch events, sort by `detected_at` desc, pick hero = first event in last 48h with max `importance`. If no event in 48h, hero = latest material event overall. If zero events, hero = "Baseline established; waiting for the first signal" placeholder.
- **3.2** Filter chips (static buttons with `?pillar=` query + client JS to filter the DOM, no router needed).
- **3.3** Feed grid (main column) + `StateOfPlay` (sidebar).

## Phase 4 ‚Äî `/events/[slug]`

Dynamic route from events collection. Render title, source badge, detected_at, `what_changed`, `implication` if any, collapsible raw diff, LLM disclosure.

## Phase 5 ‚Äî `/sources/[slug]`

Dynamic route generated from `sources.yaml` (all 18 active). Shows:
- Header: display name + "last changed N days ago / M material events tracked" (from opt-out-matrix + events).
- List of events filtered by `source == slug`.
- List of snapshot files (just filename + date for v1; diff viewer is v1.1).

## Phase 6 ‚Äî `/about`

Static `.mdx` with: what this is, methodology, how to report errors, LLM disclosure.

## Phase 7 ‚Äî `/feed.xml`

Custom endpoint at `src/pages/feed.xml.ts` ‚Äî generates RSS 2.0 from events (title, link, guid, pubDate, full content:encoded).

## Phase 8 ‚Äî `/subscribe`

Static page with an email input + POST to `/api/subscribe` (implemented in Plan 3). In Plan 2 the form just posts to a stub endpoint that returns "Subscribe coming soon" ‚Äî or we can omit the handler entirely and have the form be a display-only placeholder.

## Phase 9 ‚Äî Dev server

`npm run dev -- --host` ‚Üí http://localhost:4321 for user to verify locally.

## Notes

- **Empty state matters**: after Plan 1's first run, there are ZERO events. The homepage must look intentional with zero events ("7 crawlers under watch, nothing's changed yet"). This is the MVP test.
- **Aesthetic polish** (animations, custom cursor, gradient meshes, etc.) deferred to a later frontend-design pass once shape is validated. V1 just needs to feel editorial and competent.
